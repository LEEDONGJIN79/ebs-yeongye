EBS 연계 독서 학습지 · 문제편 (A4)

이 개념, 쉽게 알아보기

여러분, 인기 게임의 서버가 사람이 너무 몰려서 "서버 점검 중"이라는 안내가 뜨거나, 큰 행사 예매 사이트가 한꺼번에 접속이 몰려 먹통이 되는 일을 본 적 있죠? 컴퓨터 한 대가 처리할 수 있는 일에는 한계가 있어서, 데이터가 많아지고 요청이 몰리면 그 한 대가 버거워하는 거예요. 이럴 때 쓰는 똑똑한 방법이 바로 샤딩(sharding)입니다.

먼저 큰 그림부터 볼게요. 우리가 어떤 사이트에 접속해 정보를 받아 오면, 그 일을 처리해 주는 컴퓨터를 서버라고 하고, 정보를 요청하는 내 컴퓨터·휴대폰을 클라이언트라고 불러요. 그런데 데이터를 비싼 컴퓨터 한 대에 몽땅 넣어 두면, 그 한 대가 고장 나거나 해킹당하면 모든 게 한꺼번에 위험해지고, 일도 느려집니다. 그래서 데이터를 여러 대의 값싼 컴퓨터에 나눠 두는 전략이 등장했는데, 이것을 분산 파일 관리 체계라고 합니다.

샤딩은 그중 한 방법이에요. 두꺼운 백과사전을 한 권으로 들고 다니는 대신 ㄱ권·ㄴ권·ㄷ권으로 나눠 책꽂이 여러 칸에 꽂아 두는 모습을 떠올려 보세요. 이렇게 데이터를 작은 덩어리로 쪼갠 것을 샤드(shard)라고 하고, 각 샤드를 서로 다른 기계에 나눠 저장하는 게 샤딩이에요. 누가 "ㄴ권에 있는 내용"을 찾으면, 그게 어느 칸(어느 기계)에 있는지 알려 주는 안내원이 필요한데, 그 역할을 하는 것이 라우터이고, 무엇이 어디 있는지 적어 둔 장부가 설정 서버입니다.

이렇게 나눠 두면 싼 기계 여러 대로 비용을 아끼고(저장소 분산), 여러 기계가 일을 나눠 해서 빨라지는(부하 분산) 좋은 점이 있어요. 하지만 좋은 점만 있는 건 아닙니다. 찾으려는 데이터가 여기저기 흩어져 있으면 끌어모으느라 오히려 느려지는 성능 병목이 생기고, 한 기계가 망가지면 그 안의 데이터를 되살리기 어려운 낮은 장애 허용성이 문제예요. 그래서 같은 데이터를 다른 기계에도 한 벌 더 베껴 두는 복제를 함께 써서 약점을 메웁니다.

한마디로: 샤딩은 "데이터를 쪼개 나눠 담아 싸고 빠르게" 만드는 방법이고, 그 대신 "흩어져서 느려지거나 잃기 쉬운" 약점은 복제로 보완해요. 이 단원은 그 원리와 장단점, 그리고 보완책을 차근차근 익히는 데 목표가 있습니다.

이 단원에서 배울 것

이 단원의 중심에는 샤딩이 있습니다. 데이터를 어떻게 나눠 담고(구조), 그래서 무엇이 좋아지고 무엇이 위험해지는지(장단점), 약점을 어떻게 메우는지(보완)를 세 세트로 나눠 살펴봅니다.

세트 2

데이터를 나눠 담는 샤딩과 베껴 담는 복제를 나란히 놓고, 두 방식의 원리와 장단점이 어떻게 갈리는지를 견줍니다.

세트 3

클라이언트의 요청이 라우터를 거쳐 알맞은 샤드에 닿기까지의 실제 동작 과정을 따라가며, 샤딩에 복제를 결합해 운용하는 모습을 살핍니다.

세 세트를 차례로 풀고 나면, 샤딩이 왜 데이터를 쪼개는지, 그 대가로 무엇을 얻고 무엇을 잃는지, 그리고 그 약점을 어떻게 메우는지가 또렷해질 거예요.

세트 1

서버는 클라이언트의 요청에 따라 데이터를 검색하거나 연산을 수행해 그 결과를 돌려주는 컴퓨터 시스템이다. 데이터가 많아지고 요청이 몰릴수록 단일 기계의 성능과 저장 용량만으로는 이를 감당하기 어렵다. 이런 한계를 넘어서기 위해 개발된 것이 데이터를 여러 기계에 나누어 관리하는 분산 파일 관리 체계이며, 샤딩은 그 한 방법이다. 샤딩은 데이터를 샤드 단위로 분할하여 서로 연결된 여러 기계에 나누어 저장하는 방법이고, 샤드란 관리하기 쉽도록 분할하여 각 기계에 저장한 데이터 세트를 가리킨다. 클라이언트의 요청이 들어오면 어떤 샤드에 해당 데이터가 있는지를 판별하고 그 샤드에서 데이터를 조회해 전달하는 라우터가 동작하며, 라우터가 이 역할을 하도록 전체 샤드의 분할 정보와 위치 정보를 저장·관리하는 설정 서버가 따로 마련된다. 설정 서버에는 각 샤드가 어떻게 이용되었는지를 보여 주는 로그 정보가 함께 저장되기도 한다.

샤딩의 성능은 클라이언트의 요청이 처리되는 속도로 정의되며, 단위 시간당 처리되는 요청의 양으로 가늠된다. 샤딩에서는 데이터의 쓰기·읽기와 연산을 각 샤드를 맡은 서로 다른 기계가 나누어 처리하므로, 요청이 여러 기계에 분산되어 작업이 신속해진다. 또한 저장 용량이 큰 고가의 기계 한 대 대신 저장 용량이 작은 저가의 기계 여러 대를 합쳐 동일한 저장 용량을 확보하므로 비용이 절감된다. 다만 하나의 요청에 필요한 데이터가 여러 샤드에 흩어져 있으면, 그것을 끌어모아 처리하는 과정에서 저성능 기계 간 통신에 따른 지연이 생겨 성능이 떨어진다. 그래서 하나의 요청으로 함께 접근되는 데이터는 가능하면 같은 샤드에 모이도록 설계하여 데이터의 지역성을 확보하는 것이 성능을 높이는 핵심이 된다.

데이터가 무단으로 변조되거나 기계적 결함으로 손상되었을 때 정상 상태로 되돌릴 수 있는 성질을 장애 허용성이라 한다. 샤드마다 서로 다른 데이터가 들어 있는 순수한 샤딩에서는, 한 기계의 데이터가 손상되면 그 샤드의 데이터를 되살릴 길이 마땅치 않아 장애 허용성이 낮다. 이를 보완하는 방법이 샤딩에 복제를 결합하는 것이다. 복제는 서로 다른 기계에 같은 내용의 데이터를 두는 분산 방법으로, 한 기계의 데이터가 손상되어도 다른 기계가 복제본으로 정상 데이터를 가지고 있어 에러를 복구할 수 있게 한다. 따라서 ㉠샤딩에 복제를 결합하면 손상된 데이터가 있더라도 복구가 용이하다.

샤딩을 설계할 때 함께 고려해야 할 세 가지 특성이 있다. 비용 절감성은 저가의 저성능 기계 여러 대로 동일한 저장 용량과 처리 능력을 확보해 비용을 줄일 수 있다는 성질이다. 처리 신속성은 요청을 여러 샤드에 분산해 단위 시간당 더 많은 요청을 신속히 처리할 수 있다는 성질로, 데이터의 지역성이 확보될수록 높아진다. 장애 허용성은 데이터가 손상되어도 정상 상태로 복구할 수 있다는 성질이다. 그런데 장애 허용성을 높이려고 복제를 결합하면, 데이터를 추가·수정·삭제하는 쓰기 작업은 원본과 모든 복제본에 일관되게 적용되어야 하므로 쓰기 처리가 느려져 처리 신속성이 낮아진다. 또 복제본을 두는 만큼 더 많은 기계가 필요해 비용 절감성도 낮아진다. 이처럼 세 특성은 서로 맞물려 있어, 한 특성을 높이면 다른 특성이 낮아지기 쉽다. 아직까지 샤딩은 비용 절감성·처리 신속성·장애 허용성을 함께 높일 수 있는 방법이 없어, 운용 목적에 따라 어느 특성을 우선할지를 선택해 설계한다.

블록체인 기술은 데이터를 블록이라는 단위로 묶어 체인 형태로 연결한 것을 여러 대의 컴퓨터에 중복 저장하는 기술이다. 체인 형태로 연결된 블록의 집합을 블록체인이라 하고, 블록체인을 저장하는 컴퓨터를 노드라고 한다. 새로 생성된 블록은 노드들에 전파된다. 노드들은 블록에 포함된 내용이 블록체인의 다른 블록에 있는 내용과 상충되지 않는지, 동일한 내용이 블록체인의 다른 블록에 이중으로 포함되어 있지 않은지 검증한다. 검증이 끝난 블록을 블록체인에 연결할지 여부는 모든 노드들이 참여하는 승인 과정을 통해 정해진다. 승인이 완료된 블록은 블록체인에 연결되고, 이 블록체인은 노드들에 저장된다. 승인 과정에는 합의 알고리즘이 사용되고, 합의 알고리즘의 예로 ‘작업증명’이 있다.

블록체인 기술의 성능은 블록체인에 데이터가 저장되는 속도로 정의되며, 단위 시간당 블록체인에 저장되는 데이터의 양으로 계산될 수 있다. 블록체인 기술은 공개형과 비공개형으로 구분된다. 비공개형은 공개형과 달리 노드 수에 제한을 두고, 일반적으로 공개형에 비해 합의 알고리즘의 속도가 빠르다. 따라서 비공개형은 승인 과정에 걸리는 시간이 짧기 때문에 성능이 높다.

데이터가 무단으로 변경되기 어렵다는 성질을 무결성이라 하는데 무결성은 블록체인 기술의 대표적인 장점이다. 특정 노드에 저장되어 있는 일부 데이터가 변경되면 변경된 블록과 그 이후의 블록들은 블록체인과의 연결이 끊어진다. 끊어진 모든 블록을 다시 연결하는 것은 승인 과정을 필요로 하기 때문에 연결을 복구하는 것은 어렵다. 즉 블록과 블록체인의 연결을 유지하면서 블록체인에 포함된 데이터를 변경하는 것이 어려우므로 블록체인 데이터는 무결성이 높다. 무단 변경과 달리, 일부 데이터가 지워져도 승인된 원래의 데이터로 복원할 때는 승인 과정이 필요하지 않다. 따라서 ㉠블록체인에 포함된 데이터는 일부가 지워지더라도 복원이 용이하다.

블록체인 기술에서 고려해야 할 세 가지 특성이 있다. 보안성은 데이터의 무단 변경이 어려울 뿐 아니라 동일한 내용의 데이터가 블록체인의 서로 다른 블록에 또는 단일 블록에 이중으로 포함되는 것이 어렵다는 성질이다. 승인 과정에 걸리는 시간이 줄거나 노드 수가 감소하면 보안성은 낮아진다. 탈중앙성은 승인 과정에 다수의 노드들이 참여하고, 특정 노드가 승인 과정을 주도하지 않는다는 성질이다. 노드 수가 감소하면 탈중앙성은 낮아진다. 확장성은 블록체인 기술이 목표로 하는 응용 분야에 적용 가능할 만큼 성능이 높고, 노드 수가 증가해도 서비스 유지가 가능하다는 성질이다. 노드 수가 증가하면 성능이 저하되므로, 확장성이 높다는 것은 노드 수가 증가하더라도 성능 저하가 크지 않다는 것을 의미한다. 그래서 기술 변화 없이 확장성을 높이고자 할 때 노드 수를 제한하는 방법이 사용되기도 한다. 노드 수를 제한하면 성능 저하를 막을 수 있기 때문이다. 아직까지 블록체인 기술은 보안성, 탈중앙성, 확장성을 함께 높일 수 있는 방법이 없어 대규모로 채택되지 못하고 있다.

서버에 저장된 데이터가 많아지고 클라이언트의 요청이 몰리면, 단일 기계로는 안전하고 신속한 처리가 어려워진다. 이런 부담을 줄이기 위해 데이터를 여러 기계에 나누어 관리하는 분산 방법이 쓰이는데, 대표적인 두 방법이 샤딩과 복제이다. 두 방법은 데이터를 여러 기계에 둔다는 점은 같지만, 무엇을 어떻게 두는지에서 갈린다.

샤딩은 데이터를 샤드 단위로 분할하여 서로 다른 기계에 나누어 저장하는 방법이다. 각 기계에는 서로 다른 내용의 샤드가 하나씩 놓이므로, 같은 데이터가 두 기계에 겹쳐 있지 않다. 클라이언트의 요청이 들어오면 라우터가 어떤 샤드에 해당 데이터가 있는지를 판별하여 그 샤드에서 데이터를 조회해 전달하며, 라우터가 이렇게 동작하도록 전체 샤드의 분할 정보와 위치 정보를 저장한 설정 서버가 함께 운영된다. 샤딩에서는 데이터의 쓰기·읽기와 연산을 각 샤드를 맡은 기계가 나누어 처리하므로 요청이 여러 기계에 분산되어 작업이 신속해지고, 저가의 기계 여러 대로 동일한 저장 용량을 확보하므로 비용이 절감된다. 다만 하나의 요청에 필요한 데이터가 여러 샤드에 흩어져 있으면 그것을 끌어모으는 데에 시간 지연이 생기는 성능 병목이 나타날 수 있다. 저성능 기계 간 통신에서 생기는 지연이 한 기계에서 통합 연산을 수행할 때의 지연보다 클 수 있기 때문이다. 그래서 한 요청으로 함께 접근되는 데이터는 가능하면 같은 샤드에 모이도록 설계하여 데이터의 지역성을 확보하는 것이 성능을 높이는 핵심이 된다.

복제는 샤딩과 달리 서로 다른 기계에 같은 내용의 데이터를 저장하는 방법이다. 데이터의 불법 변조나 결함이 발생했을 때, 정상 데이터를 다른 기계가 복제본으로 가지고 있어 에러를 복구할 수 있게 함으로써 장애 허용성을 확보한다. 또한 클라이언트의 읽기 요청은 원본과 여러 복제본에 나누어 처리할 수 있어, 읽기가 많은 상황에서 부하가 분산되는 이점이 있다. 그러나 데이터를 추가·수정·삭제하는 쓰기 작업은 원본이나 복제본 가운데 어느 하나에서만 단독으로 이루어져서는 안 되고 원본과 모든 복제본에 일관되게 적용되어야 한다. 그래서 복제본이 많아질수록 쓰기 작업의 연산 속도가 느려지는 문제가 생긴다.

샤딩과 복제는 약점이 서로 다르기 때문에, 둘을 결합하여 약점을 보완하기도 한다. 순수한 샤딩은 샤드마다 다른 데이터가 들어 있어 한 기계의 데이터가 손상되면 복구가 어려워 장애 허용성이 낮은데, 여기에 복제를 결합하면 손상된 샤드를 다른 기계의 복제본으로 되살릴 수 있다. 가령 기계 A, B, C, D에 서로 다른 샤드 s1, s2, s3, s4를 각각 저장한 뒤, s1의 복제본을 B에도 두어 B가 원본 s2와 복제본을 함께 갖게 하고, 같은 방식으로 각 샤드의 복제본을 이웃한 기계에 분산해 둔다. 이렇게 하면 어느 한 기계가 손상되어도 그 기계가 맡았던 샤드를 다른 기계의 복제본에서 되살릴 수 있다. 다만 이 결합은 복제본을 둘 기계가 더 필요해 비용이 늘고, 쓰기 작업을 원본과 복제본에 일관되게 적용해야 해 쓰기 처리가 느려진다. 결국 샤딩과 복제 중 무엇을, 어떻게 결합할지는 비용·속도·안전 가운데 무엇을 더 중시하느냐에 따라 결정된다.

  • ② 샤딩에서는 각 기계에 서로 다른 내용의 샤드가 놓여 같은 데이터가 두 기계에 겹쳐 있지 않다.

  • ④ 샤딩에 복제를 결합하면 한 기계가 손상되어도 그 샤드를 다른 기계의 복제본으로 되살릴 수 있다.
  • ⑤ 하나의 요청에 필요한 데이터가 같은 샤드에 모일수록 데이터를 끌어모으는 지연이 커진다.

일반 사용자가 디지털 카메라를 들고 촬영하면 손의 미세한 떨림으로 인해 영상이 번져 흐려지고, 걷거나 뛰면서 촬영하면 식별하기 힘들 정도로 영상이 흔들리게 된다. 흔들림에 의한 영향을 최소화하는 기술이 영상 안정화 기술이다.

영상 안정화 기술에는 빛을 이용하는 광학적 기술과 소프트웨어를 이용하는 디지털 기술 등이 있다. 광학 영상 안정화(OIS) 기술을 사용하는 카메라 모듈은 렌즈 모듈, 이미지 센서, 자이로 센서, 제어 장치, 렌즈를 움직이는 장치로 구성되어 있다. 렌즈 모듈은 보정용 렌즈들을 포함한 여러 개의 렌즈들로 구성된다. 일반적으로 카메라는 렌즈를 통해 들어온 빛이 이미지 센서에 닿아 피사체의 상이 맺히고, 피사체의 한 점에 해당하는 위치인 화소마다 빛의 세기에 비례하여 발생한 전기 신호가 저장 매체에 영상으로 저장된다. 그런데 카메라가 흔들리면 이미지 센서 각각의 화소에 닿는 빛의 세기가 변한다. 이때 OIS 기술이 작동되면 자이로 센서가 카메라의 움직임을 감지하여 방향과 속도를 제어 장치에 전달한다. 제어 장치가 렌즈를 이동시키면 피사체의 상이 유지되면서 영상이 안정된다.

렌즈를 움직이는 방법 중에는 보이스코일 모터를 이용하는 방법이 많이 쓰인다. 보이스코일 모터를 포함한 카메라 모듈은 중앙에 위치한 렌즈 주위에 코일과 자석이 배치되어 있다. 카메라가 흔들리면 제어 장치에 의해 코일에 전류가 흘러서 자기장과 전류의 직각 방향으로 전류의 크기에 비례하는 힘이 발생한다. 이 힘이 렌즈를 이동시켜 흔들림에 의한 영향이 상쇄되고 피사체의 상이 유지된다. 이외에도 카메라가 흔들릴 때 이미지 센서를 움직여 흔들림을 감쇄하는 방식도 이용된다.

OIS 기술이 손 떨림을 훌륭하게 보정해 줄 수는 있지만 렌즈의 이동 범위에 한계가 있어 보정할 수 있는 움직임의 폭이 좁다. 디지털 영상 안정화(DIS) 기술은 촬영 후에 소프트웨어를 사용해 흔들림을 보정하는 기술로 역동적인 상황에서 촬영한 동영상에 적용할 때 좋은 결과를 얻을 수 있다. 이 기술은 촬영된 동영상을 프레임 단위로 나눈 후 연속된 프레임 간 피사체의 움직임을 추정한다. 움직임을 추정하는 한 방법은 특징점을 이용하는 것이다. 특징점으로는 피사체의 모서리처럼 주위와 밝기가 뚜렷이 구별되며 영상이 이동하거나 회전해도 그 밝기 차이가 유지되는 부분이 선택된다.

먼저 k 번째 프레임에서 특징점들을 찾고, 다음 k+1 번째 프레임에서 같은 특징점들을 찾는다. 이 두 프레임 사이에서 같은 특징점이 얼마나 이동하였는지 계산하여 영상의 움직임을 추정한다. 그리고 흔들림이 발생한 곳으로 추정되는 프레임에서 위치 차이만큼 보정하여 흔들림의 영향을 줄이면 보정된 동영상은 움직임이 부드러워진다. 그러나 특징점의 수가 늘어날수록 연산이 더 오래 걸린다. 한편 영상을 보정하는 과정에서 영상을 회전하면 프레임에서 비어 있는 공간이 나타난다. 비어 있는 부분이 없도록 잘라 내면 프레임들의 크기가 작아지는데, 원래의 프레임 크기를 유지하려면 화질은 떨어진다.

샤딩은 데이터를 샤드 단위로 분할하여 서로 연결된 여러 기계에 나누어 저장하고 활용하는 방법이다. 클라이언트가 어떤 데이터를 요청하면, 그 데이터가 어느 샤드에 있는지를 ㉠판별하여 해당 샤드에서 데이터를 조회해 클라이언트에 전달하는 장치가 필요한데, 이 장치를 라우터라 한다.

라우터가 이러한 역할을 ㉡수행하도록 전체 샤드의 분할 정보와 위치 정보를 저장하고 관리하는 설정 서버가 별도로 마련된다. 설정 서버에는 어떤 데이터가 어느 샤드에 들어 있는지를 적어 둔 위치 정보가 ㉢담기며, 각 샤드가 어떻게 이용되었는지를 보여 주는 로그 정보가 함께 저장되기도 한다. 라우터는 클라이언트의 요청을 받을 때마다 설정 서버의 위치 정보를 참고하여, 요청된 데이터가 들어 있는 샤드를 ㉣찾아낸다. 한편 데이터의 손상에 대비하기 위해 샤딩에 복제를 결합하기도 한다. 복제는 서로 다른 기계에 같은 내용의 데이터를 두는 방법으로, 원본 샤드의 복제본을 다른 기계에 함께 저장해 둔다. 데이터가 손상되면 복제본을 이용해 정상 데이터로 되살릴 수 있다.

샤딩에서 데이터는 읽기 요청의 처리와 쓰기 요청의 처리가 서로 다른 방식으로 이루어진다. 읽기 요청은 데이터를 바꾸지 않으므로, 라우터는 요청된 데이터가 있는 샤드만 찾아 그 데이터를 조회해 돌려주면 된다. 데이터가 한 샤드에 모여 있으면 한 기계만 조회하면 되어 신속하지만, 여러 샤드에 흩어져 있으면 여러 기계에서 조회한 결과를 끌어모아야 해 지연이 생긴다. 그래서 한 요청으로 함께 읽히는 데이터를 같은 샤드에 모으는 데이터의 지역성이 성능을 높이는 핵심이 된다. 반면 쓰기 요청은 데이터를 바꾸므로, 복제를 결합한 경우 원본 샤드뿐 아니라 그 복제본에도 같은 변경을 일관되게 ㉤적용해야 한다. 그래서 복제본이 많을수록 쓰기 처리가 느려진다.

㉮클라이언트가 데이터를 정상적으로 읽어 오는 과정을 살펴보자. 클라이언트는 먼저 라우터에 원하는 데이터를 요청하는 질의 패킷을 보낸다. 라우터는 설정 서버의 위치 정보를 참고하여 그 데이터가 어느 샤드에 있는지를 판별한 뒤, 해당 샤드를 맡은 기계에 데이터를 조회하는 질의 패킷을 보낸다. 그 기계는 요청된 데이터를 찾아 응답 패킷으로 라우터에 돌려주고, 라우터는 이를 클라이언트에 전달한다. 만일 라우터가 위치 정보에서 해당 데이터를 찾지 못하면, 라우터는 클라이언트에 데이터를 찾을 수 없다는 응답 패킷을 보낸다.

㉯샤딩에 복제를 결합한 서버에서 쓰기 요청이 처리되는 과정은 이와 다르다. 클라이언트가 데이터를 추가·수정하는 쓰기 요청을 보내면, 라우터는 그 데이터가 저장될 원본 샤드를 판별한다. 그런 다음 라우터는 원본 샤드를 맡은 기계뿐 아니라 그 샤드의 복제본을 가진 기계에도 같은 변경을 적용하라는 패킷을 보낸다. 원본과 모든 복제본에 변경이 일관되게 적용되어야 데이터가 어긋나지 않으므로, 한 곳이라도 변경이 빠지면 안 된다. 모든 적용이 끝나야 비로소 쓰기 요청이 완료된다.

DNS(도메인 네임 시스템) 스푸핑은 인터넷 사용자가 어떤 사이트에 접속하려 할 때 사용자를 위조 사이트로 접속시키는 행위를 말한다. 이는 도메인 네임을 IP 주소로 변환해 주는 과정에서 이루어진다.

인터넷에 연결된 컴퓨터들이 서로를 식별하고 통신하기 위해서 각 컴퓨터들은 IP(인터넷 프로토콜)에 따라 ㉠만들어지는 고유 IP 주소를 가져야 한다. 프로토콜은 컴퓨터들이 연결되어 서로 데이터를 주고받기 위해 사용하는 통신 규약으로 소프트웨어나 하드웨어로 구현된다. 현재 주로 사용하는 IP 주소는 ‘***.126.63.1’처럼 점으로 구분된 4개의 필드에 숫자를 사용하여 ㉡나타낸다. 이 주소를 중복 지정하거나 임의로 지정해서는 안 되고 공인 IP 주소를 부여받아야 한다.

공인 IP 주소에는 동일한 번호를 지속적으로 사용하는 고정 IP 주소와 번호가 변경되기도 하는 유동 IP 주소가 있다. 유동 IP 주소는 DHCP라는 프로토콜에 의해 부여된다. DHCP는 IP 주소가 필요한 컴퓨터의 요청을 받아 주소를 할당해 주고, 컴퓨터가 IP 주소를 사용하지 않으면 주소를 반환받아 다른 컴퓨터가 그 주소를 사용할 수 있도록 해 준다. 한편, 인터넷에 직접 접속은 안 되고 내부 네트워크에서만 서로를 식별할 수 있는 사설 IP 주소도 있다.

인터넷은 공인 IP 주소를 기반으로 동작하지만 우리가 인터넷을 사용할 때는 IP 주소 대신 사용하기 쉽게 ‘www..’ 등과 같이 문자로 ㉢이루어진 도메인 네임을 이용한다. 따라서 도메인 네임을 IP 주소로 변환해 주는 DNS가 필요하며 DNS를 운영하는 장치를 네임서버라고 한다. 컴퓨터에는 네임서버의 IP 주소가 기록되어 있어야 하는데, 유동 IP 주소를 할당받는 컴퓨터에는 IP 주소를 받을 때 네임서버의 IP 주소가 자동으로 기록되지만, 고정 IP 주소를 사용하는 컴퓨터에는 사용자가 네임서버의 IP 주소를 직접 기록해 놓아야 한다. 인터넷 통신사는 가입자들이 공동으로 사용할 수 있는 네임서버를 운영하고 있다.

㉮사용자가 어떤 사이트에 정상적으로 접속하는 과정을 살펴보자. 웹 사이트에 접속하려고 하는 컴퓨터를 클라이언트라 한다. 사용자가 방문하고자 하는 사이트의 도메인 네임을 주소창에 직접 입력하거나 포털 사이트에서 그 사이트를 검색해 클릭하면 클라이언트는 기록되어 있는 네임서버에 도메인 네임에 해당하는 IP 주소를 물어보는 질의 패킷을 보낸다. 네임서버는 해당 IP 주소가 자신의 목록에 있으면 클라이언트에 이 IP 주소를 알려 주는 응답 패킷을 보낸다. 응답 패킷에는 어느 질의 패킷에 대한 응답인지가 적혀 있다. 만일 해당 IP 주소가 목록에 없으면 네임서버는 다른 네임서버의 IP 주소를 알려 주는 응답 패킷을 보내고, 클라이언트는 다시 그 네임서버에 질의 패킷을 보내는 단계로 돌아가 같은 과정을 반복한다. 클라이언트는 이렇게 ㉣알아낸 IP 주소로 사이트를 찾아간다. 네임서버와 클라이언트는 UDP라는 프로토콜에 ㉤맞추어 패킷을 주고받는다. UDP는 패킷의 빠른 전송 속도를 확보하기 위해 상대에게 패킷을 보내기만 할 뿐 도착 여부는 확인하지 않으며, 특정 질의 패킷에 대해 처음 도착한 응답 패킷을 신뢰하고 다음에 도착한 패킷은 확인하지 않고 버린다. DNS 스푸핑은 UDP의 이런 허점들을 이용한다.

㉯DNS 스푸핑이 이루어지는 과정을 알아보자. 악성 코드에 감염되어 DNS 스푸핑을 행하는 컴퓨터를 공격자라 한다. 클라이언트가 네임서버에 특정 IP 주소를 묻는 질의 패킷을 보낼 때, 공격자에도 패킷이 전달되고 공격자는 위조 사이트의 IP 주소가 적힌 응답 패킷을 클라이언트에 보낸다. 공격자가 보낸 응답 패킷이 네임서버가 보낸 응답 패킷보다 클라이언트에 먼저 도착하고 클라이언트는 공격자가 보낸 응답 패킷을 옳은 패킷으로 인식하여 위조 사이트로 연결된다.
